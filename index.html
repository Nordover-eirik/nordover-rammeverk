Variabler Steg 1 — sett opp først

Variabler definerer verdiene i designsystemet ditt. Endrer du én variabel, oppdateres alt som bruker den — farger, fonter, spacing — over hele nettsiden.

Kombiner klasser

V4 lar deg legge til flere Global Classes på ett element. En CTA-knapp kan ha btn-primary + btn-small. En seksjon kan ha section-pad-m + bg-light + container-full. Tenk komposisjon.

Komponenter Steg 3 — gjenbrukbare blokker

Komponenter er hele seksjoner/containere som synkroniserer globalt. Stiling er låst, men innhold (tekst, bilder, lenker) kan redigeres per instans via Component Properties.

Standardkomponenter å bygge for hvert prosjekt

Header Komponent

  • Logo (bilde-property)
  • Navigasjonslenker
  • CTA-knapp (tekst-property)
  • Mobil hamburger-meny

Footer Komponent

  • Logo + kort beskrivelse
  • Kolonner med lenker
  • Sosiale medier-ikoner
  • Copyright-tekst (tekst-property)

Hero-seksjon Komponent

  • Overskrift (heading-xl)
  • Underoverskrift (body-default)
  • CTA-knapp(er)
  • Bakgrunnsbilde/video

CTA-boks Komponent

  • Overskrift (heading-m)
  • Kort tekst
  • Knapp (btn-primary)
  • Valgfritt bakgrunnsbilde

Tjenestekort Komponent

  • Ikon/bilde
  • Tittel (heading-s)
  • Beskrivelse (body-small)
  • Lenke/knapp

Kontaktskjema Komponent

  • Overskrift + undertekst
  • Atomic Form-elementer
  • Suksess-/feilmelding (States)
  • Personvernlenke
💡 Component Properties Eksponer kun de feltene kunden trenger å redigere (tekst, bilder, lenker). Alt annet — layout, spacing, fonter — er låst i komponenten og arves globalt.

Navngivingskonvensjon Hold det konsistent

Velg ett system og bruk det på alle prosjekter. Her er formatet anbefalt for Nordover-prosjekter:

Format

/* Komponent-klasser */ btn-primary btn-secondary card-default card-hover hero-section nav-main /* Layout / utility-klasser */ flex-row flex-col flex-center gap-s gap-m section-pad-m container-boxed /* Typografi-klasser */ heading-xl heading-l heading-m body-default body-small label text-muted /* Bakgrunn-klasser */ bg-light bg-dark bg-primary

Regler

  • Alltid lowercase med bindestrek — aldri camelCase eller underscore
  • Beskrivende nok til at noen andre forstår — btn-primary ikke bp1
  • Prefiks etter kategori: btn-, card-, heading-, section-, flex-, bg-, gap-
  • Størrelser bruker xs / s / m / l / xl — ikke tallverdier i klassenavnet
  • Unngå kundenavn i klassenavn — de skal gjenbrukes på tvers av prosjekter
⚠️ Ikke gjør dette

fjordShippingButton — kundespesifikt, camelCase
section1 — sier ingenting om hva den gjør
blåKnapp — beskriver utseende, ikke funksjon (hva om fargen endres?)
ml-32 — hardkodet verdi, bruk variabel i stedet

Responsivt design V4 sitt store fortrinn

I V4 kan du justere hver eneste stilregel per enhet — du trenger ikke lenger skjule elementer og lage kopier for mobil.

✅ Gjør dette i V4

  • Endre flex-direction fra row til column på mobil
  • Juster font-size per breakpoint direkte på klassen
  • Endre padding/margin per enhet
  • Endre gap-størrelse på mobil
  • Endre grid-kolonner per enhet

❌ Slutt med dette (V3-vaner)

  • Skjule elementer på mobil + lage duplikat
  • Egne «mobil-only»-seksjoner
  • Inline custom CSS for responsive tweaks
  • Bruke separate widgets for desktop vs mobil

Responsive regler per klasse

Arbeidsflyt Fra brief til lansering

Følg denne rekkefølgen for hvert nytt kundeprosjekt. De første timene brukes på fundament — det sparer deg for dager senere.

1
Brief & kickoff Samle inn merkevarefarger, fonter, logo-filer. Definer alle Variabler i Elementor (farger, typografi, spacing). Importer fonter via Google Fonts eller custom uploads.
2
Designsystem Bygg alle Globale Klasser fra denne guiden. Tilpass verdier til kundens visuell profil. Opprett Komponenter for header, footer, CTA, kort. Test alt på én enkelt side.
3
Prototype / førsteside Bygg forsiden kun med klasser og komponenter. Bruk aldri lokal styling med mindre det er et genuint unntak. Hvis du oppdager behov for en ny klasse — lag den nå, ikke bruk lokal stil.
4
Sideutbygging Resterende sider bygges ved å gjenbruke eksisterende klasser og komponenter. Side 2, 3, 4 osv. går betydelig raskere enn side 1.
5
Kvalitetskontroll Åpne Class Manager → sjekk for ubrukte klasser, duplikater, og inkonsistente navn. Test responsivt på alle breakpoints. Sjekk ytelse (PageSpeed).
6
Overlevering til kunde Kunden redigerer kun innhold via Component Properties (tekst, bilder). Designsystemet er låst og konsistent. Dokumenter variabelnavnene ved behov.

Sjekkliste Bruk før lansering

Gå gjennom denne listen for hvert prosjekt. Klikk for å huke av (lagres ikke — bruk som visuell gjennomgang).

Designsystem

  • Alle fargevariabler er definert og brukt — ingen hardkodede hex i klasser
  • Typografi-variabler satt opp med riktige fonter og fallbacks
  • Spacing-variabler brukes konsekvent i alle klasser
  • Alle knappvarianter har hover-, focus- og active-states
  • Class Manager — ingen ubrukte eller duplikate klasser
  • Navngivingskonvensjon er fulgt konsistent — lowercase, bindestrek, beskrivende

Responsivt

  • Alle sider testet på desktop, tablet og mobil
  • Font-størrelser justert per breakpoint i klassene
  • Seksjons-padding redusert på mobil
  • Ingen skjulte duplikatelementer — alt løst med responsive stilregler
  • Bilder har riktig aspect-ratio på alle enheter

Ytelse & kvalitet

  • PageSpeed Insights score 85+ på mobil
  • Bilder optimalisert — WebP, lazy loading, riktig dimensjon
  • Ingen V3-widgets brukt der V4-element finnes
  • Ingen inline custom CSS — alt løst med klasser/variabler
  • Alle lenker fungerer og peker riktig
  • Skjema tester OK — suksess/feil-states vises riktig

Klientoverlevering

  • Component Properties er satt opp for redigerbare felt
  • Ikke-admin brukere kan ikke endre Global Classes eller Variables
  • Favicon og OG-bilder er satt opp
  • SEO-titler og meta utfylt på alle sider

Tips & vanlige feller Ting å huske

Global Class vs lokal stil

Tommelfingerregel: Hvis du setter en stil på et element og tenker «dette bruker jeg nok igjen» — lag en Global Class. Hvis du setter den samme lokale stilen på 2+ elementer, er det et tydelig signal om at det burde vært en klasse.

V3 + V4 sameksistens

For nye sider: bruk V4 Atomic Elements + klasser. For eksisterende sider du ikke vil bygge om: behold V3, legg til V4 kun for nye seksjoner. V3-widgets støtter ikke klasser/variables, men fungerer fint ved siden av.

States = hover/focus/active

Definer States på klassen, ikke på elementet. Da arver alle elementer med den klassen de samme interaktive tilstandene. hover-styling settes automatisk som focus-visible også — bra for tilgjengelighet.

Farger i Class Manager

Fargekodede indikatorer viser hvor en stil kommer fra: grønn = Global Class, rosa = lokal, grå = arvet. Klikk på indikatoren for å se kilde og løse konflikter.

⚠️ Unngå disse feilene
  • Bruke V3 Global Colors i stedet for V4 Variables — migrer til Variables
  • Lage for mange utility-klasser — hold deg til de som faktisk gjenbrukes
  • Glemme å sette responsive verdier i klassen — da ender du opp med lokal CSS
  • Navngi klasser etter utseende (blue-box) — navngi etter funksjon (card-default)
  • Ikke rydde i Class Manager — gjør det til slutt i hvert prosjekt
