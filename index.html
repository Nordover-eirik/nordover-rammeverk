Enheter rem vs px — når bruker du hva

Tommelregel: rem for alt som bør skalere med brukerens tekststørrelse, px for alt som bør forbli visuelt identisk uansett.

rem Skalerer proporsjonalt

1rem = nettleserens base font-size (vanligvis 16px). Skalerer opp hvis brukeren har satt større tekst. Bruk for:

  • Font-size
  • Padding og margin
  • Gap
  • Container max-width
  • Seksjon-spacing
  • Knapp-padding
px Alltid identisk

Absolutt verdi. Endres ikke. Bruk for:

  • Border-width (1px, 2px)
  • Box-shadow (offsets og blur)
  • Outline / focus-ring
  • Border-radius
  • Icon-størrelse
  • Divider-tykkelse, underline-offset
Unitless for line-height, em for letter-spacing

line-height: 1.6 (uten enhet) multipliseres med font-size og skalerer automatisk. letter-spacing: -0.02em er relativt til font-size — skalerer med teksten. Aldri bruk px på noen av disse.

Navngivingskonvensjon Konsistens på tvers

Samme format på alle prosjekter. Aldri endre navnene — bare variabelverdiene.

  • Beskriver funksjon, ikke utseende
  • Størrelser: 2xs / xs / s / m / l / xl / 2xl / 3xl
  • Aldri kundenavn i klassenavn

Sjekkliste Før lansering

Designsystem

  • Fargevariabler definert — ingen hardkodede hex i klasser
  • Typografi satt opp med fonter + fallbacks
  • Spacing bruker variabler
  • Knapper har states — hover, focus, active
  • Class Manager ryddet
  • Riktige enheter — rem for spacing/font, px for border/shadow/radius

Responsivt

  • Alle sider testet desktop, tablet, mobil
  • Font-size justert per breakpoint
  • Seksjon-padding redusert for tablet/mobil
  • Grid kollapser riktig
  • Touch-targets min 44×44px mobil

Ytelse

  • PageSpeed mobil 85+
  • Bilder WebP, lazy load
  • V4-elementer brukt der tilgjengelig
  • Ingen inline CSS

Overlevering

  • Lenker fungerer
  • Skjema OK
  • SEO-titler + meta
  • Favicon + OG-bilder
  • Properties eksponert for kunde
  • Rolletilgang — kunde kan ikke endre klasser/variabler
